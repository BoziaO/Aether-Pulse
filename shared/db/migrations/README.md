# Database Migrations

This folder contains database migration files generated by [Drizzle Kit](https://orm.drizzle.team/drizzle-kit).

## Usage

### Generate Migrations

To generate a new migration file after making schema changes:

```bash
# From the workspace root
pnpm --filter @workspace/db push

# Or from this directory
cd shared/db
drizzle-kit push --config ./drizzle.config.ts
```

This will:

1. Compare your current database schema with the schema defined in `src/schema/`
2. Generate a new migration file in this directory
3. Apply the migration to your database

### Force Push (Recreate Database)

If you need to completely recreate your database from scratch:

```bash
pnpm --filter @workspace/db push-force
```

Or:

```bash
drizzle-kit push --force --config ./drizzle.config.ts
```

**Warning:** This will drop and recreate your entire database, causing data loss.

## Migration Files

Each migration file should be named with a timestamp prefix (e.g., `0000_first_migration.sql`) and contain SQL
statements to transform your database schema.

## Production Deployment

For production deployments with PostgreSQL:

1. Update `drizzle.config.ts` to use PostgreSQL dialect:

   ```ts
   dialect: 'pg',
   dbCredentials: {
     url: process.env.DATABASE_URL,
   },
   ```

2. Generate migrations for PostgreSQL:

   ```bash
   pnpm --filter @workspace/db push
   ```

3. Apply migrations using your preferred migration tool (Flyway, Liquibase, etc.) or use Drizzle Kit directly.

## Current Schema

The database includes the following tables:

- **users** - User accounts with authentication info
- **rooms** - Voice/video chat rooms
- **room_members** - Room membership junction table
- **messages** - Chat messages in rooms
- **dm_conversations** - Direct message conversations
- **dm_participants** - DM conversation participants
- **dm_messages** - Direct messages
- **friendships** - User friend relationships
- **message_reactions** - Emoji reactions on messages

## Indexes

The following indexes have been added for optimal query performance:

### Users Table

- `users_username_idx` - For username lookups (login, search)
- `users_display_name_idx` - For display name searches
- `users_status_idx` - For status queries (online users, etc.)
- `users_created_at_idx` - For time-based queries and sorting

- `rooms_invite_code_idx` - For invite code lookups

- `dm_messages_created_idx` - For time-based queries

### Reactions Table

### Friendships Table

- `friendships_addressee_idx` - For addressee-based queries
- `friendships_status_idx` - For status filtering

## Best Practices

1. **Always generate migrations after schema changes** - This ensures your database stays in sync
2. **Review migration SQL** - Check the generated SQL before applying to production
3. **Test migrations locally** - Apply migrations to a local test database first
4. **Backup production data** - Always backup before running migrations in production
5. **Use transactions** - For complex migrations, wrap in transactions for atomicity

## Troubleshooting

If you encounter issues with migrations:

1. **Schema drift**: Run `drizzle-kit push --force` to recreate the database from scratch
2. **Type errors**: Ensure all your schema types are properly imported and used
3. **Connection issues**: Verify your `DATABASE_URL` environment variable is set correctly
